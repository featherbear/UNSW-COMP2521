Top tips!

Tutorial

Be warm, friendly and welcoming. Create an atmosphere that encourages contributions. You should aim for 25 students and 25 teachers, not one teacher and 24 students.

Code reviews should run each week. Choose a pair each week to present a 5-10 minute code review of an exercise from the previous week. Try to facilitate a supportive, constructive review; try not to dominate this discussion, and mediate to keep the review moving and the tone positive. Encourage students – both the reviewers and the reviewees – to blog about this activity.

Labs

  • The warmup and challenge exercises are optional, but as much as possible encourage your students to do them.

    • The warmup exercises are intended to gently introduce the topics and provide extra practise for students who need it.

    • The challenge exercises are intended for students who finish early and need extension. These may not be possible for students until after Thursday lectures.

  • Make sure to check over students’ coding style.

  • Remember, as much as possible, don’t take over a student’s workstation.

When demonstrating…

When demonstrating, try to use a similar setup to the students:

  • Use gedit to edit files.
    • Avoid using Vim, Emacs, Sublime Text, Atom, etc.
    • In a terminal, use Nano.
  • Use 3c to compile code.
    • Avoid dcc.

  • If you need a graphical interface, use VLAB, CSE’s virtual lab environment.

    • You can fetch a TigerVNC Viewer executable and run it on tute-room workstations with no problems.

    • Do not use VLAB Gateway (‘web browser’ VLAB, rather than the VNC-client VLAB). Stop students using VLAB Gateway if you see them doing so.

  • When using a shell, try to use a standard bash. Avoid using your own custom, whiz-bang setup.

After your tute…

  • Write up (on WebCMS 3) how your tute went!
